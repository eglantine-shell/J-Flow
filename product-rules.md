# 产品规则文档

本文档定义 V1 的核心业务规则，优先级高于实现细节。
开发过程中如出现“界面能做，但规则未明确”的情况，应以本文档为准，不允许擅自补充产品逻辑。

---

## 一、基础概念

### 1. 决策库条目
指用户平时主动添加、长期保存在系统中的项目。
它具有完整属性，例如：
- 活动类型
- 时间场景
- 兴趣程度
- 是否必要
- 是否需要准备
- 重复规则
- 是否分次

它是“长期存在的模板型条目”。

### 2. 当日事项
指某一天实际出现在计划或 todo 列表中的事项实例。
来源可能包括：
- 自动生成的命中当日规则事项
- 在决策模式中被选入当天计划的非必要事项
- 用户手动添加的临时事项

### 3. 时间场景
时间场景是 tag，不是互斥库。
一个条目可以同时属于多个时间场景。
例如：
- 周末
- 假期
- 晚上

V1 中，时间场景仅用于筛选，不用于打分。
同时，部分时间场景可用于映射条目的时段归属：
- V1 不对用户自定义时间场景做自动语义判断
- 只有少数系统内置标签参与白天/晚上映射
- 当前映射规则为：
  - 白天：上午、下午、白天
  - 晚上：晚上、夜间
- 其他标签（包括用户自定义标签）不参与映射
- 不参与映射的条目默认归为 `default_day`

### 4. 活动类型
活动类型为每条决策库条目的主类型。
每条条目必须且只能有一个主活动类型。

## 二、初始化设置规则

### 1. 首次使用必须完成初始化
首次打开应用时，用户必须完成基础设置后才能进入主界面。

初始化设置包括：
- 时间场景
- 活动类型

### 2. 系统提供默认选项
时间场景和活动类型都应提供少量默认值，用作引导。
用户可在此基础上新增、删除。

### 3. 删除时间场景规则
允许删除时间场景。
删除时弹出确认提示：

“将从所有条目中移除此标签，不删除条目本身。”

确认后：
- 从所有已存在条目中移除该标签
- 不删除条目
- 不删除历史实例

### 4. 删除活动类型规则
允许删除活动类型，但若仍有决策库条目在使用该类型，则不允许删除。
提示文案应明确说明：
- 该活动类型仍被使用
- 需先修改或删除相关条目后再删除该活动类型

---

## 三、添加决策库条目规则

用户在添加区新增条目时，字段如下：

### 必填字段
- 活动类型
- 具体内容

### 可选/可配置字段
- 日期（仅在必要事项或重复条目时显示）
- 时间场景（可多选）
- 兴趣程度（1/2/3，默认 2）
- 是否必要（默认否）
- 是否需要准备（默认否）
- 准备备注
- 重复规则（不重复 / 每天 / 每周 / 每月 / 每年）
- 是否分次（默认否）

### 规则说明
1. 时间场景可为空，但为空时该条目在决策筛选中的适配性较弱
2. 必要事项不参考兴趣程度参与推荐竞争
3. 需要准备在 V1 中不影响排序，仅作为提示展示
4. 分次事项不预设总次数，每次完成后由用户设置百分比进度
5. 分次事项在后续推进时以“继续该项目”的方式存在
6. 非必要且不重复的条目不显示日期，也不记录日期
7. 必要事项或重复条目显示日期，默认值为添加当天
8. 对于重复条目，重复规则以该日期为锚点
9. V1 当前只支持“日历型重复（calendar-based recurrence）”，不支持“间隔型重复（interval-based recurrence）”
10. `daily / weekly / monthly / yearly` 在 V1 中都按日历型重复理解，不按“每隔 x 天 / 每隔 x 周 / 每隔 x 月”理解

## 四、决策模式规则

### 1. 决策模式按“天”工作
用户在决策模式中查看的是某一天的完整计划表，而非单独查看某一个时段。
页面仅区分两个时段语境：
- 白天
- 晚上

### 2. 自动进入当天计划的事项
以下事项在该日自动进入当天计划：
- 所有必要事项
- 所有按日期规则触发的非必要重复任务
- 命中当天日期的非必要一次性条目

它们不需要通过“增加安排”触发。

显示要求：
- 必要事项应在 UI 上与普通事项做出区分
- 可通过底色或标记体现“自动出现/必要”
- 自动进入当天计划的非必要条目按普通事项样式展示

### 2.1 时段归属规则
- 若条目具备可明确映射到白天的时间场景，则归为“白天”
- 若条目具备可明确映射到晚上的时间场景，则归为“晚上”
- 若无法明确映射，则默认归为“白天”

### 2.2 自动排序规则
当天计划中的自动排序顺序如下：
1. 明确映射到白天的条目
2. 默认归到白天的条目
3. 明确映射到晚上的条目

### 2.3 用户调整排序后的晚间语境规则
若用户主动调整排序，把一个“默认归白天”的条目拖到至少一个“明确映射到晚上”的条目之后，则：
- 该条目视为进入晚间语境
- 该条目切换为晚间样式
- 该条目后续应按夜间条目对待

若该条目之后再被拖回白天区域，则：
- 该条目恢复为白天语境
- 该条目恢复为白天样式

### 3. 用户手动增加安排
用户可以在当前时段语境中点击“增加安排”。
流程如下：
1. 选择活动类型
2. 系统从决策库中筛选符合条件的非必要事项
3. 给出默认推荐项
4. 用户可接受推荐，也可展开排序列表自行挑选

### 4. 非必要事项筛选规则
筛选时只考虑：
- 活动类型匹配
- 时间场景部分匹配即可

V1 不做：
- 场景命中数量打分
- 摩擦程度判断
- 智能学习用户偏好

### 5. 非必要事项排序规则
在筛选结果中按以下顺序排序：
1. 兴趣程度
2. 加入时间

加入时间排序方向为用户设置项：
- 正序
- 倒序

补充规则：
- 若某模板当天已在其他时段语境出现，则该模板仍可出现在候选列表中，但应排在所有“当天未出现过”的候选项之后
- 该类模板不作为默认推荐项
- 同一模板在同一天的同一时段语境内，不允许重复选入
- 自动进入当天计划的事项不参与推荐竞争

### 6. 推荐逻辑
推荐逻辑即“排序后的第一项”。
默认推荐展示给用户。
若用户不满意，可点击查看完整排序列表并自行选择。

## 五、Todo 模式规则

### 1. Todo 模式展示某一天的完整待办
展示内容包括三类来源：
- 自动进入当天计划的事项
- 决策模式中选入的非必要事项
- 手动添加的临时事项

补充说明：
- 自动进入当天计划的事项中，必要事项以必要事项身份展示
- 自动进入当天计划的非必要条目按普通事项身份展示

### 1.1 排程样式规则
- 颜色属于排程条目本身，而不是时段容器
- 白天条目使用半透明浅色样式
- 晚上条目使用半透明深色样式

### 2. 临时事项规则
临时事项只有一行文字内容，不具备完整决策库属性。
它不属于决策库，也不影响决策库状态。

### 3. 临时事项不回写
若用户手动添加一个与决策库内容相似甚至相同的临时事项，例如“读《abc》”，则：
- 视为临时事项
- 不影响决策库中对应项目的状态
- 不推进分次项目进度
- 不改变重复任务状态

这是产品规则，不视为 bug。

## 六、分次任务规则

### 1. 分次任务定义
分次任务指“适合多次推进完成”的项目。
例如：
- 一本书
- 一篇长文章
- 一次大整理

### 2. V1 的分次方案
V1 采用自由进度推进方案：
- 添加时只标记“是否分次”
- 不要求用户预设总次数
- 不显示固定的 1/n、2/n

### 3. 完成与推进的区别
对于非分次任务：
- 点击完成，表示该事项完成

对于分次任务：
- 点击完成，不直接代表项目整体结束
- 应提供“推进进度”的概念，由用户拖动进度条或输入百分比
- 后续再次推荐时，应表现为“继续该项目”
- 百分比进度范围为 0-100
- 其中 0-99 表示继续中，100 表示该实例整体完成

---

## 七、重复任务规则

### 1. 重复规则类型
- 不重复
- 每天
- 每周
- 每月
- 每年

### 2. V1 的 recurrence 语义
V1 当前支持的 recurrence 只有：
- `none`
- `daily`
- `weekly`
- `monthly`
- `yearly`

这些 recurrence 在 V1 中都属于“日历型重复”：
- `daily`：从锚点日期开始，每天命中
- `weekly`：每周命中锚点对应的星期
- `monthly`：每月命中锚点对应的日号
- `yearly`：每年命中锚点对应的月 + 日

补做、延后做、提前做都不会改变未来锚点。
例如“每周三买牛奶”，即使本周四才补做，下周仍按周三命中。

### 3. V1 不支持的 recurrence 语义
V1 不支持以下“间隔型重复”：
- 每隔 x 天
- 每隔 x 周
- 每隔 x 月

这类规则与当前日历型重复是两套不同语义：
- 日历型重复：未来触发点由锚点对应的日历位置决定
- 间隔型重复：未来触发点会受实际完成时间影响

后续若需要支持“每 x 天 / 每 x 周 / 每 x 月”，应作为另一套 recurrence 结构扩展，而不是直接复用当前 `daily / weekly / monthly / yearly` 的语义。
- 每天
- 每周
- 每月
- 每年

### 2. 重复任务按日期触发
重复任务是否出现在某一天，完全按日期规则决定，而不是按用户是否进入对应时段决定。

补充规则：
- 所有必要事项都会自动进入当天计划，无论是否重复
- 所有在该日期被日期规则触发的非必要重复任务，也会自动进入当天计划
- 命中当天日期的非必要、一次性条目，也会自动进入当天计划
- `daily` 表示从锚点日期开始每天都生效
- 日期规则决定非必要重复任务“今天该不该出现”
- 每个条目都以自身日期作为规则锚点
- 若条目日期为 `2026-04-17`：
  - 选择“每周”，表示每周五生效
  - 选择“每月”，表示每月 17 号生效
  - 选择“每年”，表示每年 4 月 17 日生效
- 若用户希望修改重复锚点，应通过日期输入控件修改条目日期
- 必要属性决定事项是否属于必要事项身份，并影响展示与推荐逻辑
- 自动进入当天计划的必要事项以必要事项样式展示，不参与推荐竞争
- 自动进入当天计划的非必要条目按普通事项样式展示
- 非必要、不重复、且未命中当天日期的事项不自动进入当天计划，仍通过决策模式选入或手动添加
- 允许用户在决策模式中将未来日期的一次性非必要条目提前手动选入更早的一天
- 若该条目被提前完成，则：
  - 该条目的本次计划视为已完成
  - 取消其原目标日期的自动出现
  - 这不等同于将 `TaskTemplate.isArchived` 设为 `true`

补充说明：
- `TaskTemplate.isArchived` 仅表示用户主动停用该模板，不再参与决策与自动生成
- “提前完成后不再在原目标日期自动出现”表达的是该条目本次触发已被消费，不是整个模板被归档
- 只要某条 `DayPlanItem` 在完成后会抵消模板于 `targetDate` 当天的自动出现，就应视为消费了该次触发机会
- 这包括：
  - 当天正常自动触发并完成
  - 提前手动完成未来 `targetDate` 的事项
- 临时事项或不抵消任何目标日期触发机会的普通手动事项，不应视为消费触发机会

### 3. 新周期生成规则
进入新周期时：
- 自动生成该周期的新实例
- 上一周期未完成的实例记为“过期未完成”
- 不滚动延续到新周期

### 4. 重复任务与分次任务可叠加
例如：
“整理书架”一次两天，每月一次，则它可以是每月重复，同时也是分次项目。
在这种情况下：
- 每个周期生成一个新的周期实例
- 每个实例内部可持续推进
- 周期结束未完成则记为过期未完成

## 八、需要准备规则

### 1. V1 中的作用
“需要准备”在 V1 中仅用于提示，不影响排序、不影响筛选。

### 2. 展示要求
若事项标记为需要准备，则在展示时应有明显提示，例如：
- 图标
- 备注
- 小标签

### 3. 未来预留
数据结构需为未来的“前置提醒”功能预留扩展空间，但 V1 不实现提醒逻辑。

---

## 九、时间场景推导规则

### 1. `sceneTagIds` 来源
V1 中，决策推荐所使用的 `sceneTagIds` 由系统自动推导，不需要用户手动额外选择。

### 2. 推导依据
V1 当前只从以下信息推导：
- 日期
- 当前白天 / 晚上语境

例如：
- 周中 / 周末
- 白天 / 晚上

### 3. 不参与自动推导的场景
天气相关 tag 在 V1 中不自动推导，也不参与实际命中。

---

## 十、不做事项

V1 明确不做以下功能：
- 天气 API 接入
- 按命中程度打分
- 按摩擦程度自动过滤
- 自动计算任务时长
- 自动排满整天日程
- 多设备同步
- 登录系统
- 推送提醒
- 智能学习用户偏好
- 模糊搜索（后续可能加入）

---

## 十、冲突处理原则

若出现以下情况，统一按此原则处理：

### 1. 临时事项与决策库条目重复
不合并，不回写，各自独立。

### 2. 删除时间场景
只移除标签，不删除条目。

### 3. 删除活动类型
若仍被使用，则禁止删除。

### 4. 必要事项与普通事项冲突
必要事项优先显示，普通事项不覆盖必要事项。

### 5. 规则未定义的新情况
在未更新本文档前，不允许开发阶段自行补充新业务规则。
应先补规则，再实现。
