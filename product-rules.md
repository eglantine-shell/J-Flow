# 产品规则文档

本文档定义当前有效的产品规则。

当前处于 V2 方向迁移阶段：
- V1 已完成可试用 MVP，相关记录继续保留
- V2 的核心调整来自试用反馈
- 若文档之间有冲突，以本文档为准
- 本轮只更新概念和后续任务，不改现有实现

---

## 一、V2 核心定位

J-Flow V2 的产品本体为：

普通 Todo List + 种草清单 + 有空时帮你拔草。

这意味着：
- 普通日常事务优先走 Todo
- 原“决策库”的本质改写为“种草清单”
- 原“决策推荐”的本质改写为“拔草推荐”
- 产品不再强调“所有事务都先进入库，再进入计划”

---

## 二、核心概念

### 1. Todo
Todo 是主页面主体，也是用户处理日常事务的默认入口。

Todo 负责承载：
- 普通临时事项
- 当天明确要做的事
- 必要事项
- 重复事项实例
- 从种草清单中被选入或推荐加入的事项

规则：
- 新增事项时，默认按临时事项 / 普通待办处理
- 用户不应被强制先走种草录入流程

### 2. 种草清单
种草清单承载的是“先记住，等有空再做”的内容。

例如：
- 书单
- 影单
- 探店清单
- 旅游想法
- 逛公园清单
- 待处理的家务死角清单

规则：
- 种草清单是独立清单
- 它不再承担普通待办职责
- 它更接近长期保留、等待合适语境再被挑选的事项池

### 3. 拔草推荐
拔草推荐指用户在某种“有空”语境下，从种草清单中挑选当前适合做的内容。

规则：
- 拔草推荐是增强能力，不是主页面唯一入口
- 它服务于“我现在有空，可以做点什么”
- 它不要求所有事项都经过推荐流程

### 4. 种草分类
原 `activityTypes` 在 V2 的产品表达上，更接近“种草分类”。

例如：
- 阅读
- 观影
- 旅游

规则：
- 底层数据结构暂时仍沿用 `activityTypes`
- 但产品表述应尽量使用“种草分类”或接近该含义的表达

### 5. 有空就做场景
原 `sceneTags` 在 V2 的产品表达上，更接近“有空就做场景”。

例如：
- 工作日晚上
- 周末
- 长假

规则：
- 底层数据结构暂时仍沿用 `sceneTags`
- 但产品表述应尽量使用“有空就做场景”或接近该含义的表达

---

## 三、初始化规则

### 1. 首次使用仍需完成初始化
首次打开应用时，用户仍需先完成基础初始化。

初始化包含两部分：
- 种草清单方向的默认分类
- 有空就做场景

### 2. 初始化页文案方向
初始化页后续应调整为以下表达：

标题：
- 哪怕你是一个100%的J人，想必也会有……

第一组：
- 种草清单
- 用于引导用户理解自己会长期积累想做内容

第二组：
- 有空就做
- 用于引导用户定义“什么样的空闲，适合拿来拔草”

### 3. 默认项
默认种草分类：
- 阅读
- 观影
- 旅游

默认有空就做场景：
- 工作日晚上
- 周末
- 长假

### 4. 初始化校验规则
仍需至少保留：
- 一个分类
- 一个场景

---

## 四、主页面规则

### 1. 主页面结构
V2 主页面后续应改为单页结构：
1. 头部
2. 日期
3. Todo
4. 种草

### 2. 不再强调双模式
页面不再以“决策模式 / Todo 模式”作为核心产品表达。

说明：
- V1 中的双模式是既有实现
- V2 文档口径中，应将其视为待迁移旧结构
- 后续实现目标是单页整合，而不是继续强化双模式心智

### 3. Todo 为默认入口
用户打开主页面后，应优先进入 Todo 使用语境。

规则：
- Todo 是当天事务主视图
- 用户记录简单事项时，不应被引导先进入种草或推荐流程

---

## 五、事项新增规则

### 1. 默认新增逻辑
新增事项时，默认为临时事项 / 普通待办。

这是 V2 的重要方向调整。

### 2. 加入临时事项与加入安排需要融合
V2 后续实现中，Todo 添加入口应整合：
- 直接新增临时事项
- 从种草中选择
- 推荐拔草

用户应先获得一个轻量录入入口，再按需展开更多选项。

### 3. 展开更多选项
在默认轻量录入之外，可展开：
- 从种草中选择 / 推荐拔草
- 必要事项
- 重复规则
- 需要准备
- 分次事项

规则：
- 默认形态优先轻量
- 高级选项按需展开
- 不应要求普通待办一开始就填写完整模板属性

---

## 六、Todo 规则

### 1. Todo 的职责
Todo 用于展示和处理某一天真正要做的事项。

包括但不限于：
- 临时事项
- 必要事项
- 重复事项实例
- 从种草清单中加入的事项

### 2. 临时事项规则保留
V1 关于临时事项的核心规则继续保留：
- 临时事项不回写种草清单
- 临时事项不自动推进种草条目状态
- 临时事项与库中相似内容不自动合并

### 3. 白天 / 晚上语境保留
V2 继续保留白天 / 晚上语境。

说明：
- 这是当前产品中仍然有价值的轻量时段表达
- 但它服务于展示和推荐，不应重新演变成复杂排程系统

---

## 七、种草清单规则

### 1. 种草清单不是普通待办池
只有适合“以后有空再做”的内容，才适合进入种草清单。

例如：
- 想看的书
- 想看的电影
- 想去的地方
- 想慢慢推进的整理事项

不建议：
- 把所有当天就要处理的琐事都塞进种草清单

### 2. 种草条目仍可保留扩展属性
虽然 V2 弱化“全量模板化”，但种草条目仍可继续承载现有属性能力，例如：
- 分类
- 场景
- 是否必要
- 重复规则
- 需要准备
- 分次事项

说明：
- 这代表底层结构可暂时复用
- 不代表 UI 和新增入口仍要把这些字段全部强塞给用户

---

## 八、必要事项、重复、准备、分次

以下 V1 规则在 V2 暂时继续保留，不在本轮重写其底层定义：
- 必要事项
- 日历型重复规则
- 需要准备
- 分次事项

说明：
- 这些能力仍属于事项表达的一部分
- 本轮只调整产品叙事和入口顺序
- 不改现有业务实现

---

## 九、数据结构策略

### 1. 现阶段策略
V2 暂时尽量复用现有数据结构，除非后续实现中发现必须调整。

### 2. 当前明确保留
- `activityTypes`
- `sceneTags`
- 现有事项层与实例层结构

### 3. 当前明确不做
- 本轮不改数据模型
- 本轮不新增新的底层业务表意字段
- 本轮不因为概念改名而立即推翻既有结构

---

## 十、V1 概念迁移说明

以下是当前明确的概念迁移：

- `决策库` → `种草清单`
- `决策推荐` → `拔草推荐`
- `活动类型` → 产品表达上更接近“种草分类”
- `时间场景` → 产品表达上更接近“有空就做场景”
- `决策模式 / Todo 模式` → 后续迁移为单页中的不同区域，而不是双模式主结构

同时，以下 V1 旧思路降级：
- “所有事务都进入决策库”
- “普通待办也要经过模板 / 推荐 / 选入流程”

---

## 十一、不做事项

本轮不做：
- 不改源码
- 不改数据模型
- 不改业务逻辑实现
- 不做 UI
- 不引入新功能实现

若后续实现阶段遇到规则空白：
- 先补本文档
- 再做实现

---

## 十二、冲突处理原则

### 1. 文档冲突
若文档间表述冲突，以本文档为准。

### 2. 概念迁移与底层字段不一致
允许“产品表达已经 V2 化，但底层字段仍沿用 V1 命名”暂时并存。

### 3. 后续发现结构不够用
先记录问题，再评估是否需要改 `data-model.md` 与实现，不得在实现中自行扩写规则。
