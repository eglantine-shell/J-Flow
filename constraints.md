# V1 约束文档

本文档用于明确本项目第一阶段“不做什么”，防止开发过程无边界膨胀。

---

## 一、产品边界

### 1. V1 只做单人使用
不考虑多人协作，不考虑共享，不考虑家庭或团队场景。

### 2. V1 只做本地网页版本
- 运行于浏览器
- 使用本地存储
- 不做登录
- 不做后端
- 不做云同步

### 3. V1 以“可用、稳定、好理解”为优先
优先确保：
- 规则稳定
- 交互顺畅
- 数据结构合理
- 便于未来迭代

不追求：
- 功能铺满
- 一次做成完整版
- 复杂智能化

---

## 二、明确不做的功能

### 1. 不接天气 API
顶部天气图标仅保留 UI 占位。
V1 不做：
- 定位
- 天气接口
- 天气对推荐逻辑的影响

### 2. 不做场景命中打分
V1 中时间场景仅用于筛选，不用于：
- 命中数比较
- 权重打分
- 推荐值计算

### 3. 不做摩擦判断
“需要准备”仅作提示展示，不参与：
- 筛选
- 排序
- 自动过滤

### 4. 不做自动整天排程
V1 不要求用户输入事项时长。
系统不自动将一天排满，不计算总时长，不做时间块冲突检测。

### 5. 不做智能学习
V1 不根据用户历史行为自动学习偏好，不自动调整推荐逻辑。

### 6. 不做模糊搜索
V1 不实现：
- 模糊匹配
- 同义词
- 智能搜索联想

后续版本可考虑加入。

### 7. 不做提醒与通知
V1 不做：
- 前置提醒
- 浏览器通知
- 定时提醒
- 重复提醒

### 8. 不做复杂进度系统
V1 支持“分次任务”的继续推进概念，但不做：
- 自定义阶段拆分器
- 自动估算剩余进度

### 9. 不做数据同步与备份
V1 默认数据只保存在本地浏览器。
若用户清除浏览器缓存，数据可能丢失。
后续可考虑增加导出/导入能力。

### 10. 不做 App 原生能力
V1 不实现：
- iOS/Android 原生壳
- 推送
- 小组件
- 系统级分享接入

### 11. 不做间隔型重复
V1 不实现：
- 每隔 x 天
- 每隔 x 周
- 每隔 x 月

当前 `daily / weekly / monthly / yearly` 一律按“日历型重复”理解：
- `daily`：从锚点日期开始每天命中
- `weekly`：每周命中锚点对应的星期
- `monthly`：每月命中锚点对应的日号
- `yearly`：每年命中锚点对应的月 + 日

若后续要支持 interval-based recurrence，应单独扩展 recurrence 结构，不能直接改写当前字段语义。

---

## 三、实现约束

### 1. 不要为了“以后也许会用”过度设计
虽然需为未来留接口，但 V1 应保持实现轻量。
不要引入明显超出当前需求的架构复杂度。

### 2. 不要擅自新增产品规则
若发现实现中存在规则空白：
- 先记录问题
- 再补文档
- 再实现

不能由开发阶段自行决定。

### 3. 不要把模板和实例混在一起
这是当前项目最重要的数据约束之一。
若建模混乱，后续重复任务、分次任务、临时事项都会出问题。

### 4. 不要让临时事项影响决策库
临时事项仅服务于当天，不参与任何库状态推进。

### 5. 不要把“需要准备”偷偷做成排序条件
V1 明确禁止该逻辑。

---

## 四、可预留但不实现的扩展点

可以在数据结构或组件结构上预留，但 V1 不做：

- 天气接入
- 模糊搜索
- 前置提醒
- 数据导出/导入
- 智能学习
- 多设备同步
- App 化
