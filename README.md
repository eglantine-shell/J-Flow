# 决策型 Todo List：J-FLOW

一个面向高计划性用户的 all in one todo list 网页应用。

它不是普通的“记录待办”，而是一个带有决策机制的待办系统：
用户平时将想做的事情加入“决策库”，并为它们设置活动类型、适用时间场景、兴趣程度、是否必要、是否需要准备、是否重复等属性；在需要安排某一天时，系统会根据当天条件筛选并给出推荐选项，帮助用户更流畅地完成“今天做什么”的决策。

## 产品定位

这是一个为“极致 J 人”设计的工具。

适合以下用户：
- 对自己的生活习惯和偏好较了解
- 喜欢提前规划
- 平时会积累“想看、想做、想去”的内容
- 不满足于普通 todo list 的线性记录方式
- 希望把“收藏”与“安排”连接起来

当前版本不试图解决的问题：
- 不试图服务所有用户
- 不做复杂项目管理
- 不做自动排程
- 不做自动学习用户偏好
- 不做社交协作
- 不做多设备同步（V1）

## V1 核心能力

### 1. 初始化设置
首次打开时，用户需要完成基础设置：
- 时间场景（系统内置 + 自定义）
- 活动类型（系统内置 + 自定义）

时间场景允许重叠，更接近 tag，而不是互斥分类。
例如：
- 周中 / 周末 / 假期
- 上午 / 中午 / 下午 / 晚上
- 晴天 / 雨天

活动类型为每条任务的主类型。
例如：
- 学习
- 工作
- 外出
- 家务
- 阅读
- 观影
- 写作

### 2. 添加决策库条目
用户可以在主界面下部添加条目，字段包括：
- 活动类型
- 具体内容
- 适用时间场景（可多选）
- 兴趣程度（默认 2，可调为 1/2/3）
- 是否必要
- 是否需要准备
- 准备备注
- 重复规则（不重复 / 每天 / 每周 / 每月 / 每年）
- 是否分次
- 若分次，则该项目以“可持续推进”的形式存在，不预设总次数

### 3. 决策模式
用户选择某一天，页面展示当天完整的计划表。
页面中用四个浅色色块表示一天中的时间区间：
- 0-6
- 6-12
- 12-18
- 18-24

系统会先自动放入按日期触发的必要事项/重复事项。
用户再基于某个时间区块，点击“增加安排”，选择活动类型，系统从决策库中筛选符合条件的候选项，并给出默认推荐。

筛选逻辑：
1. 必要事项不参与推荐竞争，而是自动出现
2. 非必要事项只做部分匹配筛选，不按命中程度打分
3. 在筛选结果内，按兴趣程度排序
4. 兴趣程度相同时，按加入时间排序（正序或倒序可配置）

### 4. Todo 模式
Todo 模式展示某一天的完整待办清单。
来源包括：
- 自动出现的必要/重复任务
- 在决策模式中被选入当天计划的非必要任务
- 用户在 todo 模式中手动加入的临时事项

三类来源在数据结构中需要区分。

### 5. 分次任务推进
V1 采用“自由进度推进”方案：
- 用户添加条目时只标记“是否分次”，不预设总次数
- 分次项目在完成时，用户可以标记当前项目本次完成的百分比
- 单次完成不直接视为项目整体完成，除非用户将进度标记为百分百
- 系统在后续推荐进度大于0、小于百分百的分次项目时，将其视为“继续该项目”

### 6. 周期任务
重复任务按日期规则触发。
进入新周期时：
- 自动生成新实例
- 上一周期未完成的实例记为“过期未完成”
- 不延续到新周期

## 当前技术边界

V1 为单人、本地网页应用：
- 数据保存在浏览器本地
- 不做登录
- 不做后端
- 不做多设备同步
- 不接天气 API，仅预留结构
- 不做推送提醒

## 项目目标

先做一个真正可用、可发布、逻辑稳定的网页版本。
如果产品验证成功，再考虑：
- 模糊搜索
- 智能推荐优化
- 提醒机制
- 多设备同步
- App 化

## 当前前端技术栈

Task 1 当前采用以下轻量方案作为项目底座：
- `React 19`：单页应用视图层
- `TypeScript`：类型约束
- `Vite`：开发与构建工具
- `React Router`：页面路由与初始化流程切换
- `Zustand`：本地 UI / 会话状态管理
- `Dexie`：浏览器本地存储访问层（基于 IndexedDB）
- `Zod`：后续数据校验与存储边界校验
- `Vitest + Testing Library`：基础单元测试与组件测试

本阶段仅完成项目初始化与目录结构搭建，不实现业务规则。

## 当前目录结构建议

```text
src/
  app/          应用入口、路由、壳层
  components/   通用 UI 组件
  db/           本地存储访问层
  features/     按业务领域划分的功能目录
  lib/          通用常量与工具
  mocks/        mock 数据预留
  pages/        路由页面
  store/        全局状态
  styles/       全局样式与设计变量
  tests/        测试初始化
  types/        核心类型定义
```

## 启动方式

当前项目现在以 `pnpm` 作为本地与 CI 的统一包管理器。
待安装 Node.js 与 pnpm 后，可使用以下命令启动：

```bash
pnpm install
pnpm run dev
```

## GitHub Pages 部署

当前项目已按 Vite 官方推荐方式适配 GitHub Pages 项目路径部署：
- Vite `base` 已配置为 `/J-Flow/`
- GitHub Actions 会在默认分支 `main` 收到 push 后自动构建并部署
- 构建产物目录为 `dist/`
- workflow 使用 `pnpm install --frozen-lockfile` 与 `pnpm run build`

首次启用时，请在 GitHub 仓库网页中完成：
1. 打开 `Settings > Pages`
2. 在 `Build and deployment` 中将 `Source` 设为 `GitHub Actions`

部署成功后，预期访问地址为：
- `https://eglantine-shell.github.io/J-Flow/`
