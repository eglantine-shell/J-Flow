# 开发任务清单

本文档用于指导 Codex 分阶段实现项目。
要求每次只处理一个明确任务，不跨任务擅自扩写。

---

## 第一阶段：项目底座

### Task 1：初始化项目
目标：
- 创建前端项目基础结构
- 确定技术栈
- 配置基础开发环境

要求：
- 使用适合单页应用的现代前端方案
- 支持本地状态管理
- 支持本地存储
- 保留清晰的目录结构

交付物：
- 可运行项目
- 基础目录结构
- 基础路由或主页面骨架
- README 中补充技术栈说明

---

## 第二阶段：数据模型与存储

### Task 2：实现核心数据模型
目标：
- 根据 `data-model.md` 建立类型定义
- 明确模板层与实例层

要求：
- 不得将模板与实例混用
- 必须保留事项来源字段
- 必须保留周期实例层

交付物：
- 类型定义文件
- mock 数据
- 数据校验辅助函数（如需要）

### Task 3：实现本地存储层
目标：
- 建立统一的数据读写入口

要求：
- 本地存储结构与 `AppData` 一致
- 支持初始化默认值
- 支持版本升级预留
- 支持条目日期字段的默认写入
- 支持重复锚点基于条目日期计算

交付物：
- storage 模块
- 初始化函数
- 基础增删改查接口

---

## 第三阶段：首次初始化流程

### Task 4：实现首次初始化页
目标：
- 支持用户完成首次基础设置

要求：
- 系统提供默认时间场景和默认活动类型
- 支持新增、删除
- 至少保留一个时间场景和一个活动类型
- 完成后写入 `initialized = true`
- 首次进入主页面后，应以当天日期作为默认计划视图日期

交付物：
- 初始化页 UI
- 初始化数据落地逻辑
- 首次进入判断逻辑

---

## 第四阶段：主页面骨架

### Task 5：实现主页面基础布局
目标：
- 搭建主页面四大区域

要求：
- 顶部信息区
- 模式切换区
- 主内容区
- 底部添加区

交付物：
- 主页面骨架
- 日期切换
- 模式切换
- 天气图标预留位

---

## 第五阶段：决策库条目录入

### Task 6：实现底部添加区表单
目标：
- 允许用户向决策库添加条目

要求：
- 支持条件日期输入：必要事项或重复条目显示日期并默认当天，非必要且不重复时不记录日期
- 支持活动类型选择
- 支持内容输入
- 支持时间场景多选
- 支持兴趣程度设置
- 支持必要开关
- 支持需要准备开关与备注
- 支持重复规则
- 支持分次开关
- 重复条目以该日期作为重复锚点
- `daily` 需按“从锚点日起每天生效”解释

交付物：
- 可提交表单
- 数据写入决策库
- 表单默认值逻辑

### Task 7：实现决策库条目管理
目标：
- 查看已有决策库条目
- 支持编辑与删除

要求：
- 删除条目应有确认
- 编辑后更新模板数据
- 不直接改动历史实例

交付物：
- 条目管理视图或弹窗
- 编辑功能
- 删除功能

---

## 第六阶段：决策模式

### Task 8：实现决策模式页面
目标：
- 展示某一天的完整计划表

要求：
- 页面不再分四个时间区块，而是按单列计划区展示白天 / 晚上语境
- 可显示自动生成事项
- 支持在当前计划语境中“增加安排”

交付物：
- 决策模式基础 UI
- 白天 / 晚上语境下的计划区布局
- 自动事项显示占位

### Task 9：实现自动入计划事项生成逻辑
目标：
- 根据日期自动生成当天应出现的事项实例

要求：
- 所有必要事项都自动进入当天计划
- 按日期规则触发的非必要重复任务自动进入当天计划
- 命中当天日期的非必要、一次性条目也自动进入当天计划
- 非必要、不重复、且未命中当天日期的事项不自动进入当天计划
- 必要属性影响展示身份与推荐竞争
- 新周期自动生成新实例
- 旧周期未完成实例记为过期未完成
- 条目时段归属按“明确白天 / 默认白天 / 明确晚上”确定
- 自动排序顺序为：明确白天 -> 默认白天 -> 明确晚上
- 仅系统内置白天/晚上标签参与自动映射

交付物：
- 自动生成逻辑
- 周期实例状态更新逻辑
- 页面展示联动

### Task 10：实现推荐逻辑
目标：
- 在用户点击“增加安排”后给出推荐项和候选列表

要求：
- 只对非必要事项生效
- 先按活动类型匹配
- 再按系统根据“日期 + 白天/晚上语境”自动推导出的时间场景做部分匹配筛选
- 按兴趣程度排序
- 兴趣相同按加入时间排序
- 当天已在另一个时段语境出现的模板排在候选列表最末
- 当天已在另一个时段语境出现的模板不作为默认推荐项
- 允许将未来日期的一次性非必要条目提前手动选入更早的一天

交付物：
- 推荐函数
- 推荐弹窗/面板
- 候选列表 UI

### Task 11：实现决策选入逻辑
目标：
- 将用户选择的事项加入当天对应语境位置

要求：
- 生成 `DayPlanItem`
- 来源标记为 `decision_selected`
- 同一模板在同一天的同一时段语境内不允许重复选入
- 同一模板在同一天的不同语境内允许再次选入
- 默认白天条目在拖入晚间后需切换为晚间语境；拖回白天后需恢复白天语境

交付物：
- 选入逻辑
- 计划表更新
- 与 todo 模式共享实例数据

---

## 第七阶段：Todo 模式

### Task 12：实现 Todo 模式页面
目标：
- 展示某一天完整待办清单

要求：
- 展示三类来源事项
- 能区分自动进入当天计划的事项中“必要展示”和“普通展示”
- 能区分白天样式与晚上样式
- 支持完成事项
- 支持删除临时事项
- 支持查看准备备注
- 提前完成未来日期的一次性非必要条目后，需反映其原目标日期不再自动出现

交付物：
- Todo 列表 UI
- 来源区分显示
- 完成/删除交互

### Task 13：实现临时事项快速添加
目标：
- 在 todo 模式中快速加入一行文字事项

要求：
- 临时事项不回写决策库
- 临时事项不推进任何模板/周期状态

交付物：
- 快速添加输入区
- 临时事项创建逻辑
- 删除逻辑

---

## 第八阶段：分次任务处理

### Task 14：实现分次任务推进逻辑
目标：
- 支持分次事项处于“继续中”

要求：
- 分次任务点击完成时，不直接视为整体结束
- 应允许用户将其标记为继续中
- 后续推荐时仍能继续出现
- 需支持实例级 `progressPercent`，范围为 `0-100`
- `0-99` 视为继续中，`100` 视为该实例整体完成

交付物：
- 分次状态处理逻辑
- Todo/决策模式展示适配

---

## 第九阶段：设置与规则补齐

### Task 15：实现设置页
目标：
- 允许用户管理基础设置与排序偏好

要求：
- 管理时间场景
- 管理活动类型
- 设置加入时间排序方向

交付物：
- 设置页/弹窗
- 排序偏好写入
- 删除规则校验

---

## 第十阶段：稳定性与验证

### Task 16：补齐边界测试
目标：
- 验证关键规则不出错

至少覆盖以下场景：
1. 删除时间场景后，原条目仍保留
2. 删除仍被使用的活动类型时被阻止
3. 临时事项不影响库状态
4. 重复任务跨周期生成新实例
5. 旧周期未完成实例记为过期
6. 命中当天规则的事项自动进入当天计划
7. 分次任务不会被一次完成直接终结
8. 决策模式选入后能在 todo 模式中看到

交付物：
- 测试清单
- 手动验证记录
- 如有条件可加入基础单元测试

---

## 每轮开发的统一要求

每次让 Codex 执行任务时，都要求它同时输出：
1. 改了哪些文件
2. 为什么这样实现
3. 覆盖了哪些边界情况
4. 还有哪些未解决风险
